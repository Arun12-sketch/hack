# 유니버설 스마트 컨트랙트

> [!danger]
> 이 섹션에서 제공되는 모든 정보와 코드 스니펫은 교육 목적으로만 제공되며, 프로덕션 환경에 직접 배포하기 위한 것이 아닙니다.

## 유니버설 스마트 컨트랙트란?

**유니버설 스마트 컨트랙트**(USC)는 기존 스마트 컨트랙트에 크로스체인 기능을 확장하는 어댑터 역할을 합니다. 컨트랙트가 **Creditcoin 탈중앙화 오라클**을 통해 외부 블록체인의 데이터를 *조회*, *검색*, *검증*할 수 있게 해줍니다.

토큰 전송이나 특정 자산에 좁게 초점을 맞추는 전통적인 옴니체인 또는 크로스체인 솔루션과 달리, USC는 **범용 실행 레이어**를 제공합니다. 이를 통해 컨트랙트는 *핵심 로직을 다시 작성할 필요 없이* 외부에서 검증된 데이터에 따라 동작할 수 있습니다. 개발자는 기술 스택에 USC를 도입함으로써 자신의 컨트랙트를 원활한 크로스체인 데이터로 구동되는 *유니버설* 컴포넌트로 변환할 수 있으며, 여러 블록체인 간의 새로운 상호운용성 패턴을 구현할 수 있습니다.

## USC 컨트랙트 아키텍처

USC 컨트랙트는 크로스체인 증명을 검증하고 비즈니스 로직을 실행합니다. **DApp 비즈니스 로직 컨트랙트**는 Creditcoin에 배포되어 DApp의 상태와 비즈니스 로직을 포함하는 컨트랙트입니다. 예제 구현(`SimpleMinterUSC`)에서는 비즈니스 로직(ERC20 토큰 발행)이 USC 컨트랙트 자체에 직접 통합되어 있습니다. 이러한 결합 패턴은 간단한 사용 사례에 적합하지만, 더 복잡한 DApp의 경우 개발자는 관심사를 분리하여 별도의 컨트랙트를 배포할 수 있습니다: 핵심 USC 책임을 처리하는 USC 컨트랙트와 검증 성공 후 USC 컨트랙트가 호출하는 별도의 비즈니스 로직 컨트랙트입니다. 두 패턴 모두 유효하며, 선택은 DApp의 복잡성과 요구사항에 따라 달라집니다. 비즈니스 로직 컨트랙트는 USC 컨트랙트가 제공하는 검증된 크로스체인 데이터를 기반으로 실행되므로 USC 아키텍처의 일부로 간주됩니다.

## USC 작동 방식

USC 컨트랙트는 **네이티브 쿼리 검증자 프리컴파일**(주소 `0x0FD2`)을 사용하여 크로스체인 트랜잭션 데이터를 검증합니다. 이는 머클 증명과 연속성 증명의 동기식 검증을 제공하는 내장 런타임 컴포넌트입니다. USC 컨트랙트는 크로스체인 데이터를 처리하기 전에 증명을 검증하기 위해 `verify()` (또는 대안으로 `verifyAndEmit()`) 함수를 직접 호출하여 통합됩니다. 검증이 완료되면 USC 컨트랙트는 검증된 트랜잭션 바이트에서 직접 트랜잭션 및 이벤트 데이터를 추출하고 DApp 전용 비즈니스 로직을 실행합니다.

**주요 특성:**

* **동기식 검증**: 증명이 동일한 트랜잭션 내에서 검증되며, 비동기 처리 없음
* **직접 데이터 추출**: 트랜잭션 및 이벤트 데이터가 검증된 트랜잭션 바이트에서 직접 추출됨
* **재실행 방지**: USC 컨트랙트는 중복 처리를 방지하는 메커니즘을 구현
* **네이티브 속도 실행**: 프리컴파일이 최적의 성능을 위해 네이티브 Rust 코드로 실행됨

> [!danger]
> 검증 시스템은 트랜잭션이 성공했는지 여부를 검증하지 ***않습니다***. 트랜잭션이 블록에 포함되었는지, 그리고 해당 블록이 실제로 확인된 소스 체인의 일부인지만 검증합니다. 따라서 트랜잭션의 성공 여부는 스마트 컨트랙트 자체에서 검증해야 합니다.

## 핵심 USC 컨트랙트 패턴

일반적인 USC 컨트랙트는 다음 패턴을 따릅니다:

1. 오프체인 워커로부터 **증명 및 트랜잭션 데이터 수신**
2. 중복 처리를 방지하기 위한 **재실행 방지 구현**
3. 증명을 동기식으로 검증하기 위해 **네이티브 쿼리 검증자 프리컴파일 호출**
4. 검증된 트랜잭션 바이트에서 **트랜잭션/이벤트 데이터 추출**
5. 검증된 데이터를 기반으로 **비즈니스 로직 실행**

### 예제 USC 컨트랙트

완전한 USC 구현은 [`SimpleMinterUSC.sol`](https://github.com/gluwa/usc-testnet-bridge-examples/blob/main/contracts/sol/SimpleMinterUSC.sol)을 참조하세요. 이 컨트랙트는:

* 오프체인 워커로부터 증명 및 트랜잭션 데이터 수신
* `processedQueries` 매핑을 사용한 재실행 방지 구현
* 네이티브 쿼리 검증자 프리컴파일을 사용한 증명 검증
* 트랜잭션 유형 및 영수증 상태 검증 (성공해야 함)
* `EvmV1Decoder`를 사용하여 검증된 트랜잭션 바이트에서 이벤트 데이터 추출
* 소스 체인에서 소각 이벤트가 검증되면 토큰을 발행하는 동일 컨트랙트 내에서 비즈니스 로직(ERC20 토큰 발행) 실행

**주요 함수 시그니처:**

```solidity
function mintFromQuery(
    uint64 chainKey,
    uint64 blockHeight,
    bytes calldata encodedTransaction,
    bytes32 merkleRoot,
    INativeQueryVerifier.MerkleProofEntry[] calldata siblings,
    bytes32 lowerEndpointDigest,
    bytes32[] calldata continuityRoots
) external returns (bool success)
```

### DApp 비즈니스 로직 컨트랙트

DApp 비즈니스 로직 컨트랙트는 DApp의 상태와 비즈니스 로직을 포함하여 Creditcoin에 배포된 스마트 컨트랙트입니다. USC 컨트랙트가 제공하는 검증된 크로스체인 데이터를 기반으로 실행되기 때문에 USC 아키텍처의 일부로 간주됩니다.

예제 구현(`SimpleMinterUSC`)에서 비즈니스 로직은 USC 컨트랙트에 직접 통합되어 있습니다. 이 컨트랙트는:

* DApp 상태 저장 (예: ERC20을 통한 토큰 잔액)
* DApp 전용 로직 구현 (예: 토큰 발행)
* 크로스체인 증명 검증 및 트랜잭션 내용 검증 직후 즉시 비즈니스 로직 실행
* 입력 검증 및 상태 업데이트

### 트랜잭션 데이터 추출

검증이 성공하면 USC 컨트랙트는 트랜잭션 내용 검증 프로세스의 일부로 `encodedTransaction` 바이트에서 트랜잭션 및 이벤트 데이터를 추출합니다. 트랜잭션 인코딩은 다음을 포함하는 결정론적 형식을 따릅니다:

* **트랜잭션 필드**: 유형, 체인 ID, 논스, 발신자 주소, 수신자 주소, 값 등
* **영수증 필드**: 상태, 사용된 가스, 로그(이벤트)
* **이벤트 데이터**: 트랜잭션 영수증 로그의 토픽 및 데이터

USC 컨트랙트는 `EvmV1Decoder`와 같은 라이브러리를 사용하여 비즈니스 로직에 필요한 특정 이벤트나 트랜잭션 필드만 선택적으로 추출할 수 있습니다. 이러한 선택적 추출을 통해 USC 컨트랙트는 전체 트랜잭션 구조를 디코딩하지 않고도 비즈니스 로직에 필요한 특정 이벤트나 트랜잭션 필드를 효율적으로 검증할 수 있습니다.

### 쿼리 처리 흐름

오프체인 워커가 소스 체인 트랜잭션에 대한 증명 데이터를 제공할 때:

1. **워커가 증명 생성 API 서버를 사용하여 증명 생성**
2. **워커가 증명 및 인코딩된 트랜잭션 데이터와 함께 USC 컨트랙트 호출**
3. **USC 컨트랙트가 네이티브 쿼리 검증자 프리컴파일을 사용하여 증명을 동기식으로 검증**
4. **USC 컨트랙트가 검증된 트랜잭션 바이트에서 데이터 추출**
5. **USC 컨트랙트가 동일 트랜잭션 내에서 즉시 비즈니스 로직 실행**

이 모든 것이 단일 트랜잭션 내에서 동기식으로 발생합니다 - 비동기 쿼리 처리나 결과 저장이 없습니다.

### USC 컨트랙트 구현 예제

다음 섹션들은 [`SimpleMinterUSC.sol`](https://github.com/gluwa/usc-testnet-bridge-examples/blob/main/contracts/sol/SimpleMinterUSC.sol)을 기반으로 한 완전한 USC 컨트랙트 구현을 분석합니다.

#### 컨트랙트 구조

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.23;

import {ERC20} from "@openzeppelin/contracts/token/ERC20/ERC20.sol";
import {EvmV1Decoder} from "./EvmV1Decoder.sol";

contract SimpleMinterUSC is ERC20 {
    INativeQueryVerifier public immutable VERIFIER;
    mapping(bytes32 => bool) public processedQueries;

    // ... 나머지 컨트랙트
}
```

**주요 컴포넌트:**

* **ERC20 상속**: 컨트랙트는 결합 패턴을 사용합니다 - USC(증명 검증)이면서 동시에 비즈니스 로직 컨트랙트(`ERC20` 토큰 발행 로직)입니다
* **VERIFIER**: 주소 `0x0FD2`에 있는 네이티브 쿼리 검증자 프리컴파일에 대한 불변 참조
* **processedQueries**: 동일 트랜잭션의 중복 처리를 방지하는 재실행 방지용 매핑

#### 프리컴파일 인터페이스 및 헬퍼 라이브러리

```solidity
interface INativeQueryVerifier {
    struct MerkleProofEntry {
        bytes32 hash;
        bool isLeft;
    }
    struct MerkleProof {
        bytes32 root;
        MerkleProofEntry[] siblings;
    }
    struct ContinuityProof {
        bytes32 lowerEndpointDigest;
        bytes32[] roots;
    }

    function verifyAndEmit(
        uint64 chainKey,
        uint64 height,
        bytes calldata encodedTransaction,
        MerkleProof calldata merkleProof,
        ContinuityProof calldata continuityProof
    ) external view returns (bool);
}

library NativeQueryVerifierLib {
    address constant PRECOMPILE_ADDRESS = 0x0000000000000000000000000000000000000FD2;

    function getVerifier() internal pure returns (INativeQueryVerifier) {
        return INativeQueryVerifier(PRECOMPILE_ADDRESS);
    }
}
```

**설명:**

* **INativeQueryVerifier**: 프리컴파일의 검증 함수와 증명 구조체를 정의하는 인터페이스
* **NativeQueryVerifierLib**: 주소 `0x0FD2`에 있는 프리컴파일에 쉽게 접근할 수 있게 해주는 헬퍼 라이브러리
* **MerkleProofEntry**: 머클 증명 경로의 단일 노드를 나타냄 (해시 값 및 위치)
* **ContinuityProof**: 체크포인트에서 연속성을 증명하는 블록 증명 체인 포함

#### 생성자 및 초기화

```solidity
constructor() ERC20("Mintable (TEST)", "TEST") {
    // 헬퍼 라이브러리를 사용하여 프리컴파일 인스턴스 가져오기
    VERIFIER = NativeQueryVerifierLib.getVerifier();
}
```

**설명:**

* 이름과 심볼로 ERC20 토큰 초기화
* `VERIFIER` 불변 변수를 프리컴파일 인스턴스로 설정
* 프리컴파일 주소는 상수이며 항상 사용 가능

#### 재실행 방지

```solidity
mapping(bytes32 => bool) public processedQueries;
```

**설명:**

* **processedQueries**: 처리된 트랜잭션을 추적하기 위해 트랜잭션 키를 불리언 값에 매핑

#### 트랜잭션 인덱스 계산

```solidity
function _calculateTransactionIndex(INativeQueryVerifier.MerkleProofEntry[] memory proof)
    internal pure returns (uint256 index)
{
    index = 0;
    for (uint256 i = 0; i < proof.length; i++) {
        if (proof[i].isLeft) {
            index |= 1 << i;
        }
    }
    return index;
}
```

**설명:**

* **\_calculateTransactionIndex**: 머클 증명 경로에서 트랜잭션 인덱스를 도출
  * 증명 경로는 머클 트리에서의 위치를 인코딩
  * 각 `isLeft` 플래그는 형제 노드가 왼쪽(1)인지 오른쪽(0)인지를 나타냄
  * 인덱스는 리프에서 루트로 경로를 순회하여 계산됨

#### 증명 검증

```solidity
function _verifyProof(
    uint64 chainKey,
    uint64 blockHeight,
    bytes calldata encodedTransaction,
    bytes32 merkleRoot,
    INativeQueryVerifier.MerkleProofEntry[] calldata siblings,
    bytes32 lowerEndpointDigest,
    bytes32[] calldata continuityRoots
) internal view returns (bool verified) {
    INativeQueryVerifier.MerkleProof memory merkleProof =
        INativeQueryVerifier.MerkleProof({root: merkleRoot, siblings: siblings});

    INativeQueryVerifier.ContinuityProof memory continuityProof =
        INativeQueryVerifier.ContinuityProof({
            lowerEndpointDigest: lowerEndpointDigest,
            roots: continuityRoots
        });

    // 포함 증명 검증
    verified = VERIFIER.verifyAndEmit(chainKey, blockHeight, encodedTransaction, merkleProof, continuityProof);

    return verified;
}
```

**설명:**

* 제공된 컴포넌트에서 `MerkleProof` 및 `ContinuityProof` 구조체 구성
* 주소 `0x0FD2`에서 프리컴파일의 `verifyAndEmit()` 함수를 동기식으로 호출
* 머클 증명(트랜잭션 포함)과 연속성 증명(블록 증명 체인)이 모두 유효하면 `true` 반환; 실패 시 되돌림 (검증 실패 시 트랜잭션 되돌림)
* 검증 성공 시 `TransactionVerified` 이벤트 발생
* 검증은 동일 트랜잭션에서 발생 - 비동기 처리 없음

#### 메인 진입점: mintFromQuery

```solidity
function mintFromQuery(
    uint64 chainKey,
    uint64 blockHeight,
    bytes calldata encodedTransaction,
    bytes32 merkleRoot,
    INativeQueryVerifier.MerkleProofEntry[] calldata siblings,
    bytes32 lowerEndpointDigest,
    bytes32[] calldata continuityRoots
) external returns (bool success) {
    // 머클 증명 경로에서 트랜잭션 인덱스 계산
    uint256 transactionIndex = _calculateTransactionIndex(siblings);

    // 쿼리가 이미 처리되었는지 확인
    bytes32 txKey;
    {
        assembly {
            let ptr := mload(0x40)
            mstore(ptr, chainKey)
            mstore(add(ptr, 32), shl(192, blockHeight))
            mstore(add(ptr, 40), transactionIndex)
            txKey := keccak256(ptr, 72)
        }
        require(!processedQueries[txKey], "Query already processed");
    }

    // 먼저 증명 검증
    bool verified = _verifyProof(
        chainKey, blockHeight, encodedTransaction, merkleRoot, siblings,
        lowerEndpointDigest, continuityRoots
    );
    require(verified, "Verification failed");

    // 쿼리를 처리됨으로 표시
    processedQueries[txKey] = true;

    // 다음으로 트랜잭션 내용 검증
    bool valid = _validateTransactionContents(encodedTransaction);
    require(valid, "Transaction contents validation failed");

    // 비즈니스 로직 실행 (토큰 발행)
    _mint(msg.sender, MINT_AMOUNT);

    emit TokensMinted(address(this), msg.sender, MINT_AMOUNT, txKey);

    return true;
}
```

**설명:**

* **매개변수**: 오프체인 워커로부터 모든 증명 컴포넌트 및 트랜잭션 데이터 수신
* **트랜잭션 인덱스 계산**: `_calculateTransactionIndex()`를 사용하여 머클 증명 경로에서 트랜잭션 인덱스 계산
* **트랜잭션 키 생성**: 가스 효율성을 위해 어셈블리를 사용하여 `chainKey`, `blockHeight`, `transactionIndex`로 고유 키 생성
* **재실행 방지**: 이 트랜잭션이 이미 처리되었는지 확인
* **증명 검증**: `_verifyProof()`를 호출하여 머클 및 연속성 증명을 동기식으로 검증
* **상태 업데이트 (재실행 방지)**: `processedQueries` 매핑에 트랜잭션을 처리됨으로 표시
* **트랜잭션 내용 검증**: 트랜잭션 유형 및 영수증 상태를 확인하여 트랜잭션 내용 검증
* **비즈니스 로직 실행**: 검증 통과 시 비즈니스 로직 실행 (토큰 발행)
* **이벤트 발생**: 트랜잭션 세부 정보와 함께 `TokensMinted` 이벤트 발생

#### 트랜잭션 데이터 추출

컨트랙트에는 `encodedTransaction` 바이트에서 트랜잭션 데이터를 추출하고 검증하는 헬퍼 함수가 포함되어 있습니다:

```solidity
function _validateTransactionContents(bytes memory encodedTransaction)
    internal pure returns (bool valid)
{
    // 트랜잭션 유형 검증
    uint8 txType = EvmV1Decoder.getTransactionType(encodedTransaction);
    require(EvmV1Decoder.isValidTransactionType(txType), "Unsupported transaction type");

    // 영수증 상태 디코딩 및 검증
    EvmV1Decoder.ReceiptFields memory receipt = EvmV1Decoder.decodeReceiptFields(encodedTransaction);
    require(receipt.receiptStatus == 1, "Transaction did not succeed");

    // 전송 이벤트 찾기 및 검증
    EvmV1Decoder.LogEntry[] memory transferLogs =
        EvmV1Decoder.getLogsByEventSignature(receipt, TRANSFER_EVENT_SIGNATURE);
    require(transferLogs.length > 0, "No transfer events found");

    // 원래 발신자 가져오기
    EvmV1Decoder.CommonTxFields memory txFields = EvmV1Decoder.decodeCommonTxFields(encodedTransaction);

    // 발신자로부터의 실제 소각 전송이 있는지 확인
    bool found = _processTransferLogs(transferLogs, txFields.from);
    require(found, "No valid burn transfer found");

    return true;
}
```

**설명:**

* `EvmV1Decoder` 라이브러리를 사용하여 트랜잭션 바이트 디코딩
* 트랜잭션 유형 및 영수증 상태 검증
* `Transfer` 이벤트 시그니처와 일치하는 이벤트 로그 추출
* 소각 전송이 발생했는지 검증 (주소 < 128로의 전송)
