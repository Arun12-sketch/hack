> 이 섹션에서 제공되는 모든 정보와 코드 스니펫은 교육 목적으로만 제공되며, 프로덕션 환경에 직접 배포하기 위한 것이 아닙니다.

## Creditcoin 오라클 디자인 패턴

*Creditcoin 오라클을 **사용할 수 있는** 방법 vs **사용해야 하는** 방법.*

크로스체인 DApp은 Universal Smart Contract를 사용하여 최대한의 유연성을 제공하며, 소스 체인에서 실행 체인으로 데이터를 안전하게 이동할 수 있습니다.

하지만 대부분의 프로젝트는 특정 패턴을 따르는 것이 가장 좋습니다.

### 소스 체인 데이터 흐름 패턴

> [!note] 참고
> 이 패턴들은 [[USC/DApp-빌더-인프라/인프라-개요|소스 체인 스마트 컨트랙트]]로 들어오고 나가는 데이터 흐름을 관장합니다.

모든 비즈니스 로직은 Creditcoin에 있어야 합니다. 소스 체인 컨트랙트는 필요한 데이터와 함께 이벤트를 발생시키는 데 집중해야 합니다.

소스 체인의 로직은 *최소한*으로 유지하고 싶습니다!

1. 사용자가 소스 체인 스마트 컨트랙트를 호출합니다.
2. 소스 체인 컨트랙트가 하나 이상의 이벤트를 발생시킵니다.

그게 전부입니다!

### 실행 체인 데이터 흐름 패턴

> [!note] 참고
> 이 패턴들은 [[USC/DApp-빌더-인프라/유니버설-스마트-컨트랙트|Universal Smart Contract]]로 들어오고 나가는 데이터 흐름을 관장합니다.

Universal Smart Contract 실행을 *원활하게* 만들고 싶습니다!

1. 오프체인 워커가 소스 체인 이벤트를 감지합니다.
2. 워커가 Creditcoin에서 블록 증명을 기다립니다.
3. 워커가 API를 통해 머클 증명과 연속성 증명을 생성합니다.
4. 워커가 증명과 데이터를 포함하여 USC 컨트랙트를 호출합니다.
5. USC 컨트랙트가 증명을 동기적으로 검증합니다.
6. USC 컨트랙트가 즉시 비즈니스 로직을 실행합니다.

## 모범 사례

1. 모든 관련 오라클 이벤트를 발생시키는 단일 소스 체인 컨트랙트를 사용하세요.
2. 각 오라클 쿼리 유형에 대해 명확한 이벤트를 사용하세요.
3. 크로스체인 기능을 시작한다는 것을 명확히 나타내는 이벤트 이름을 사용하세요.
4. 표준 Transfer 이벤트와 같은 일반적인 이벤트는 피하세요.
5. 필요한 모든 데이터를 발생시킨 이벤트에 포함하세요.
