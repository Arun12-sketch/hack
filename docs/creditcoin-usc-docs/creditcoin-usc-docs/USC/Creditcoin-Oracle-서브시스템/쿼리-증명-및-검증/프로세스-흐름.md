# 프로세스 흐름

## 개요

쿼리-증명-검증 프로세스는 네 가지 주요 단계로 구성됩니다:

1. 쿼리 단계: 검증 대상 트랜잭션 식별
2. 증명 생성 단계: 머클 증명 및 연속성 증명 생성
3. 검증 단계: 증명의 암호학적 검증
4. 데이터 추출 단계: 검증된 바이트에서 트랜잭션 데이터 추출

"각 단계는 이전 단계를 기반으로 하여, 소스 체인 트랜잭션에서 Creditcoin 스마트 컨트랙트 실행까지 신뢰가 필요 없는 검증 체인을 생성합니다."

### 시퀀스 다이어그램

머메이드 시퀀스 다이어그램은 다음 구성 요소 간의 전체 프로세스 흐름을 보여줍니다:
- 빌더/사용자
- 소스 체인
- 증명 생성 API 서버
- Attestation 팔레트 (스토리지)
- USC 컨트랙트
- 네이티브 프리컴파일

다이어그램은 초기 블록 데이터 가져오기부터 증명 생성, 검증, 최종 비즈니스 로직 실행까지 세 가지 주요 단계(쿼리 준비, 검증, 데이터 추출)에 걸쳐 11개의 번호가 매겨진 단계를 보여주며, 구성 요소 간의 상호 작용을 상세히 설명합니다.

## 단계 1: 쿼리 단계

쿼리는 증명이 필요한 내용을 지정합니다:

* 소스 체인: 트랜잭션이 발생한 블록체인 (`chainKey`로 식별)
* 블록 높이: 트랜잭션이 포함된 블록
* 트랜잭션: 검증할 특정 트랜잭션 (트랜잭션 인덱스 또는 해시로 식별)

예시: "이더리움 메인넷의 블록 18,000,000에서 인덱스 5에 있는 트랜잭션이 실제로 발생했음을 증명하라."

쿼리 정보는 다음 용도로 사용됩니다:

* 소스 체인에서 트랜잭션 위치 찾기
* 가져와야 할 소스 체인 블록 결정
* 연속성 증명에 필요한 증명서(attestation) 식별

## 단계 2: 증명 생성 단계

"증명 생성 API (또는 직접 증명 생성)는 트랜잭션이 합법적임을 함께 증명하는 두 가지 상호 보완적인 증명을 생성합니다."

### 2.1 머클 증명 생성

API는 소스 체인 RPC 노드에서 지정된 높이의 블록을 요청합니다. "블록의 모든 트랜잭션은 해시되어 머클 트리를 형성하며, 머클 루트는 블록 헤더에 저장됩니다." 머클 증명은 다음으로 구성됩니다:

* 머클 루트 (블록 헤더에서)
* 위치 정보가 포함된 형제 해시 배열
* 트랜잭션 바이트 자체

"형제 해시와 트랜잭션 바이트를 제공하면 누구나 머클 루트까지의 경로를 재구성할 수 있습니다. 계산된 루트가 블록 헤더의 루트와 일치하면 해당 트랜잭션이 해당 블록에 있음이 증명됩니다."

### 2.2 연속성 증명 생성

API는 상위 (쿼리 블록 이후의 가장 낮은 증명서/체크포인트) 및 하위 (쿼리 블록 이전의 가장 높은 증명서/체크포인트) 증명서 경계를 찾고 Creditcoin 증명서 팔레트에 온체인으로 저장된 증명서를 요청합니다. 이러한 경계 사이의 소스 체인 블록을 가져오고 연속성 증명을 구성합니다.

"각 블록의 다이제스트는 다음을 사용하여 계산됩니다: `digest = hash(block_number, merkle_root, previous_digest)`. 이렇게 하면 각 블록의 다이제스트가 이전 블록에 의존하는 암호화 체인이 생성됩니다."

연속성 증명은 다음으로 구성됩니다:

* 연속성 체인이 시작되기 전 블록의 다이제스트 (`lowerEndpointDigest`)
* 머클 루트 배열 (다이제스트는 이러한 루트에서 온체인으로 계산됨)

"블록 번호는 쿼리 높이와 블록 인덱스에서 추론됩니다 (`roots[0]`은 `queryHeight`에 있음)"

"연속성 증명은 마지막으로 증명된 블록에서 쿼리 블록까지 끊김 없는 암호화 체인을 생성합니다. 이 체인의 어떤 블록이든 수정되면 모든 후속 다이제스트가 변경되어 변조를 감지할 수 있습니다."

증명 생성 API는 다음을 반환합니다:

* 머클 증명: 블록 내 트랜잭션 포함 증명
* 연속성 증명: 블록이 최종화된 체인의 일부임을 증명
* 인코딩된 트랜잭션: 전체 트랜잭션 바이트 (트랜잭션 + 영수증 데이터)

## 단계 3: 검증 단계

"오프체인 워커 (또는 사용자)는 증명 및 인코딩된 트랜잭션 바이트와 함께 USC 컨트랙트 함수를 호출합니다. 그런 다음 USC 컨트랙트는 네이티브 쿼리 검증기 프리컴파일을 호출하여 증명을 검증합니다."

### 3.1 머클 증명 검증 프로세스

* 시작: `leafHash = hash(transaction_bytes)`
* 각 형제에 대해: 형제 해시와 결합 (위치에 따라 왼쪽 또는 오른쪽)
* 최종 단계: `computedRoot == merkleRoot` (연속성 증명 루트 배열에서)

### 3.2 연속성 증명 검증 프로세스

* 연속성 체인의 각 블록에 대해: `computedDigest = hash(block_number, merkleRoot, previousDigest)`
* 검증: `computedDigest`가 연속성 증명 구조에서 계산된 다이제스트와 일치하는지 확인
* 최종 단계: `finalDigest == onChainAttestationDigest`

"검증은 동일한 트랜잭션 실행에서 동기적으로 발생합니다."

주요 특성:

* 대기 없음: 결과는 수 초 내에 사용 가능
* 원자적: 모든 검증 단계가 성공하거나 (트랜잭션 계속) 모두 실패 (트랜잭션 되돌림)
* 중간 상태 없음: 쿼리 저장 없음, 비동기 처리 없음, 최종화 대기 없음

"USC 컨트랙트는 동일한 트랜잭션에서 검증된 데이터를 즉시 사용할 수 있어, 다단계 비동기 흐름 없이 복잡한 크로스체인 로직이 가능합니다."

## 단계 4: 데이터 추출 단계

"검증이 성공한 후 USC 컨트랙트는 검증된 트랜잭션 바이트에서 필요한 데이터를 추출합니다. `encodedTransaction` 바이트에는 전체 트랜잭션 데이터가 포함되어 있습니다."

다음을 디코딩하는 데 사용할 수 있습니다:
- 트랜잭션 유형
- 공통 필드
- 유형별 필드
- 영수증 필드

데이터가 추출되면 USC 컨트랙트는:

* 추출된 데이터 검증 (예: 영수증 상태 = 성공, 예상 이벤트 발견)
* 검증된 크로스체인 데이터를 기반으로 비즈니스 로직 실행
* 컨트랙트 상태 업데이트 또는 추가 작업 트리거

**예시**: 브릿지 컨트랙트는 다음을 수행할 수 있습니다:

1. 이더리움에서 토큰이 소각되었음을 보여주는 `Transfer` 이벤트 검증
2. 이벤트에서 `from`, `to`, `value` 추출
3. `to` 주소로 Creditcoin에서 동등한 토큰 발행
