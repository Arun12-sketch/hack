# 쿼리를 위한 연속성 증명

## 개요

쿼리를 위한 연속성 증명은 쿼리된 소스 체인 블록과 Creditcoin에 저장된 온체인 증명서(attestation) 또는 체크포인트 간의 암호학적 연결을 수립합니다. 이러한 증명은 쿼리된 블록이 Creditcoin Oracle에 의해 증명되었으며 되돌릴 수 없음을 입증함으로써 신뢰가 필요 없는 크로스체인 검증을 가능하게 합니다.

**참고:** 쿼리 연속성 증명은 증명서 생성에 사용되는 것과 다릅니다—이 섹션에서는 쿼리 전용 증명을 다룹니다.

## 연속성 증명 구조

쿼리 연속성 증명은 쿼리된 블록을 Creditcoin에 저장된 증명서 또는 체크포인트 데이터에 암호학적으로 연결합니다. 증명에는 다음이 포함됩니다:

* **`lowerEndpointDigest`**: 쿼리 이전 블록(`queryHeight - 1`)의 다이제스트로, 인덱싱된 증명서 데이터에서 가져옴
* **`roots[]`**: `queryHeight`부터 증명서/체크포인트 블록까지의 머클 루트 배열

온체인 다이제스트 계산은 다음을 사용합니다: `digest[i] = hash(blockNumber[i], merkleRoot[i], digest[i-1])`

## 연속성 증명의 목적

연속성 증명은 악의적인 행위자가 조작된 머클 루트를 제시하거나, 확정되지 않은 블록에 트랜잭션이 있다고 거짓 주장하거나, 오래된 재조직된 블록을 악용하는 것을 방지합니다. 이는 다음을 통해 달성됩니다:

1. **온체인 증명서에 연결** — 쿼리된 블록이 증명된 소스 체인에 속함을 입증
2. **암호학적 보장 생성** — 다이제스트 체인이 끊어지지 않은 블록 시퀀스의 변조 방지 증거를 제공

### 쿼리 유형

**증명서 사이:** 쿼리 블록이 두 개의 일반 증명서 사이에 위치; API가 인덱싱된 증명서 데이터에서 연속성 증명을 구성하여 하위 증명서와 상위 증명서를 연결합니다.

**체크포인트 사이:** 쿼리 블록이 두 개의 체크포인트 사이에 위치; API가 해당 범위의 모든 증명서를 쿼리하여 향상된 최종성을 위해 체크포인트에서 끝나는 증명을 생성합니다.

## 연속성 증명 구성 과정

### 1단계: 구간 끝점 결정

API가 증명서/체크포인트 경계를 식별합니다:

* **쿼리 이전의 가장 높은 증명서/체크포인트 찾기** — `queryHeight`보다 작은 블록 번호를 가진 가장 최근 증명서/체크포인트를 찾음; 계산된 다이제스트를 `lowerEndpointDigest`로 가져옴
* **쿼리 이후의 가장 낮은 증명서/체크포인트 찾기** — `queryHeight`를 초과하는 블록 번호를 가진 가장 빠른 증명서/체크포인트를 식별; 이것이 증명의 상위 끝점이 됨

**예시:** 쿼리 높이가 145이고, 블록 140에 이전 증명서(다이제스트: `0xabc...`)가 있고 블록 150에 다음 증명서(다이제스트: `0xdef...`)가 있는 경우, 증명은 블록 145–150을 블록 150의 증명서에 연결해야 합니다.

### 2단계: 인덱싱된 데이터 쿼리 및 블록 가져오기

API가 Creditcoin에서 인덱싱된 증명서 데이터와 소스 체인 블록을 가져옵니다:

* **인덱싱된 증명서 데이터 쿼리** — 특정 증명서(또는 체크포인트 구간의 모든 증명서)를 가져옴; 계산된 루트와 다이제스트를 추출
* **블록 헤더 가져오기** — `queryHeight`부터 증명서/체크포인트 블록까지 소스 체인 RPC에서 헤더를 요청; 헤더에는 블록 번호, 머클 루트, 이전 해시가 포함됨
* **블록 무결성 검증** — 연속적인 체인 형성을 검증; 각 블록의 `previousBlockHash`가 이전 블록의 해시와 일치하는지 확인

### 3단계: 다이제스트 체인 구성

인덱싱된 Creditcoin 증명서 데이터에서 계산된 루트와 다이제스트를 사용합니다:

**다이제스트 계산:**
```
digest[i] = hash(blockNumber[i], merkleRoot[i], digest[i-1])
```

**과정:**
1. 블록 `queryHeight - 1`의 다이제스트로 초기화 (이전 증명서/체크포인트의 `prev_digest`와 일치; 사용 불가능한 경우 제네시스 다이제스트 사용)
2. 인덱싱된 데이터의 루트를 사용하여 증명서/체크포인트까지 각 블록에 대해 순차적으로 계산
3. 최종 다이제스트가 온체인 인덱싱된 증명서/체크포인트 다이제스트와 일치하는지 검증

### 4단계: 연속성 증명 구조 구축

증명 구조는 루트만 저장합니다; 다이제스트는 온체인에서 계산됩니다:

```rust
struct ContinuityProof {
    lowerEndpointDigest: bytes32,  // 블록 (queryHeight - 1)의 다이제스트
    roots: bytes32[]            // 머클 루트 배열
}
```

* **`lowerEndpointDigest`** — 쿼리 이전 블록의 다이제스트; 이전 증명서/체크포인트 다이제스트를 참조
* **`roots[]`** — `queryHeight` 이후의 머클 루트; 블록 번호는 `queryHeight + index`에서 파생됨

## 보안 모델

연속성 증명은 계단식 검증 방식을 사용합니다: 어떤 증명 블록을 수정하면 해당 다이제스트가 변경되고, 이는 모든 후속 다이제스트에 연쇄적으로 영향을 미쳐 온체인 증명서/체크포인트 다이제스트와 불일치가 발생하여 거부됩니다.

**공격 예시:**
```
공격자가 블록 147을 수정:
- 원본: root = 0x111, digest = 0xAAA
- 수정됨: root = 0x222, digest = 0xBBB

연쇄 효과:
- 블록 148 다이제스트 변경 (0xBBB 참조)
- 블록 149 다이제스트 변경 (업데이트된 148 참조)
- 블록 150 다이제스트 변경 (업데이트된 149 참조)

결과: 최종 다이제스트 ≠ 온체인 다이제스트 → 탐지 및 거부
```
