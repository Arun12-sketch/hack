# 증명을 위한 연속성 증명

## 개요

Creditcoin 탈중앙화 오라클은 체인 간 데이터를 안전하게 전송하기 위한 두 가지 필수 메커니즘 중 하나로 연속성 증명을 사용합니다. 이러한 증명은 증명 사이의 간격을 채워 소스 체인 블록의 끊김 없는 시퀀스를 유지합니다.

**연속성 증명이 중요한 이유:** 온체인 저장소 제약으로 인해 모든 블록에 대한 증명을 기록하는 것은 실용적이지 않습니다. 대신 시스템은 일정 간격(예: 10-100블록마다)으로 증명을 저장하며, 각 증명은 다이제스트 참조를 통해 연결됩니다. 연속성 증명은 이러한 간격을 연결하는 중간 블록 다이제스트 체인을 제공하여, 명시적으로 저장되지 않은 소스 체인 블록도 검증할 수 있게 합니다.

## 핵심 용어

- **해시(Hash):** 임의의 입력을 고정 크기 출력으로 변환하는 결정론적 암호화 함수
- **머클 트리(Merkle Tree):** 최소한의 부분집합 증명을 사용하여 데이터 무결성을 효율적으로 검증할 수 있게 하는 균형 이진 트리 구조
- **루트/머클 루트(Root/Merkle Root):** 모든 기반 데이터를 고유하게 나타내는 머클 트리 정점의 단일 해시
- **다이제스트(Digest):** 블록 번호, 머클 루트, 이전 다이제스트에서 파생된 "블록 또는 증명을 고유하게 식별하는 해시 출력"
- **이전 다이제스트(Previous Digest):** 현재 블록의 다이제스트 계산에 포함되는 이전 블록의 다이제스트

## 해싱을 통한 블록 체이닝

각 블록의 다이제스트가 이전 블록의 다이제스트를 포함하므로, 어떤 블록의 내용을 수정하면 이후 모든 블록 다이제스트에 변경이 연쇄적으로 전파됩니다. 이를 통해 "가장 최근 블록만 사용하여 체인의 어떤 부분이 변경되었는지 저렴하게 검증"할 수 있습니다.

## 증명 생성 프로세스

**1. 증명 블록 결정:** `next_attestation_block = latest_attestation_block + attestation_interval` 계산

**2. 블록 가져오기:** 증명자(Attestor)가 RPC 노드에서 소스 체인 블록을 조회

**3. 증명 프래그먼트 구성:** 각 새 다이제스트가 다음 공식을 따르는 다이제스트 시퀀싱으로 연속성 증명 구축: `digest = hash_payload(block_number, root, prev_digest)`

**4. 서명 및 제출:** 증명자가 SR25519(신원) 및 BLS(집계) 서명을 적용한 후 P2P 가십 네트워크에 제출

## 증명에 대한 연속성 증명 검증

검증 프로세스:
- 마지막으로 확정된 증명/체크포인트와 현재 증명 높이를 사용하여 간격 끝점 식별
- 끝점 사이의 블록 가져오기
- 표준화된 공식을 사용하여 다이제스트 계산
- 온체인에서 다이제스트 체인을 재구성하여 연속성 검증

`prev_digest`가 마지막으로 확정된 다이제스트와 정확히 일치하면 연속성 증명이 필요하지 않습니다. 그렇지 않으면 증명이 간격을 연결합니다.

**보안:** 악의적인 증명에 대한 주요 방어는 합의입니다 - 정당한 블록을 사용하는 정직한 증명자는 다른 증명을 생성하여 결함이 있는 버전이 정족수를 달성하는 것을 방지합니다.
