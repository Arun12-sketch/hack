# 마이그레이션 가이드

## 개요

최신 버전 v2는 v1에 비해 상당한 아키텍처 개선을 도입했습니다.

### V1 아키텍처

* **Prover 컨트랙트** - 오라클 인터페이스 역할을 하는 스마트 컨트랙트 (예: `IPublicProver`)
* **STARK 증명** - Cairo 프로그램과 Stakeware의 stone prover를 사용하여 생성된 암호화 증명
* **Prover 서비스** - 증명을 생성하는 별도의 오프체인 서비스
* **에스크로/결제 시스템** - 증명 검증이 완료될 때까지 에스크로에 결제금 보관
* **쿼리 제출 플로우** - 사용자가 쿼리 제출 → prover가 STARK 증명 생성 → 온체인에서 증명 검증

### V2 아키텍처

크게 단순화되었습니다:

* **네이티브 쿼리 검증 Precompile** - `0x0FD2`에서 직접 온체인 검증
* **STARK 증명 불필요** - 별도의 암호화 증명 없이 네이티브 검증
* **Prover 컨트랙트 불필요** - 직접 precompile 호출, 오라클 인터페이스 컨트랙트 배포 불필요
* **에스크로 시스템 불필요** - 결제 에스크로 없이 직접 검증
* **단순화된 플로우** - 즉각적인 결과와 함께 직접 검증 호출

> [!info]
> 아키텍처 세부 사항은 [[USC/개요/USC-아키텍처-개요|USC Architecture Overview]]를 참조하세요.

## 주요 변경 사항

### 1. Prover 컨트랙트에서 네이티브 Precompile로

#### V1에서는:

* 에스크로/결제 시스템이 있는 Prover 컨트랙트
* 오프체인 STARK 증명 생성
* 쿼리 제출 → 대기 → 결과 읽기 워크플로우

#### V2에서는:

* `0x0FD2`로 직접 precompile 호출
* 네이티브 검증 (STARK 증명 불필요)
* 즉각적인 검증 결과

### 2. 새로운 Precompile 주소 추가

#### V1에서는:

* `0x0FD1` (4049) - SubstrateTransferPrecompile
* `0x13B9` (5049) - SignatureVerifierPrecompile

#### V2에서는:

* `0x0FD1` (4049) - SubstrateTransferPrecompile (변경 없음)
* `0x13B9` (5049) - SignatureVerifierPrecompile (변경 없음)
* `0x0FD2` (4050) - **BlockProverPrecompile** (신규 - 네이티브 쿼리 검증기)
* `0x0FD3` (4051) - **ChainInfoPrecompile** (신규)

### 3. 인터페이스 함수 변경

#### V2 함수:

* `verifyAndEmit()` - `TransactionVerified` 이벤트를 발생시키는 상태 변경 함수
* `verify()` - 읽기 전용 검증을 위한 view 함수
* 두 함수 모두에 대한 배치 검증 오버로드

#### 새로운 이벤트 (선택 사항; `verifyAndEmit()`과 함께 발생):

```solidity
event TransactionVerified(
    uint64 indexed chainKey,
    uint64 indexed height,
    uint64 transactionIndex
);
```

> [!info]
> 새로운 인터페이스에 대한 자세한 내용은 [[USC/DApp-빌더-인프라/유니버설-스마트-컨트랙트|Universal Smart Contracts]]를 참조하세요.

## 스마트 컨트랙트 마이그레이션

*이 섹션에서는 v1에서 v2로 이전할 때 알아차릴 수 있는 모든 변경 사항에 대한 간략한 개요를 제공합니다.*

#### V1 접근 방식:

V1은 쿼리 결과를 저장하는 prover 컨트랙트를 사용합니다. 전체 플로우는 여러 파일과 툴체인에 걸쳐 여러 단계를 포함합니다:

**1단계: 결제와 함께 쿼리 제출**

오프체인 코드가 결제와 함께 prover 컨트랙트에 쿼리를 제출합니다 ([submit_query.ts](https://github.com/gluwa/ccnext-testnet-bridge-examples/blob/main/custom-contracts-bridging/scripts/submit_query.ts)에서 발췌):

```typescript
// Calculate query cost
const computedQueryCost = await ccNextPublicClient.readContract({
  address: proverContractAddress as `0x${string}`,
  abi: PROVER_ABI,
  functionName: 'computeQueryCost',
  args: [query],
});

// Submit query with payment
const { request } = await ccNextPublicClient.simulateContract({
  address: proverContractAddress as `0x${string}`,
  account: ccNextWalletClient.account,
  abi: PROVER_ABI,
  functionName: 'submitQuery',
  args: [query, ccNextWalletClient.account?.address!],
  value: computedQueryCost,
});

const txHash = await ccNextWalletClient.writeContract(request);
```

**2단계: Prover가 STARK 증명을 생성하고 제출할 때까지 대기**

Prover 컨트랙트가 `QuerySubmitted` 이벤트를 발생시킵니다. 오프체인 prover들이 이 이벤트를 수신하고, STARK 증명을 생성한 후 컨트랙트에 다시 제출합니다. 이 단계는 오프체인에서 비동기적으로 발생합니다.

**3단계: `QueryProofVerified` 이벤트 수신**

오프체인 코드가 증명이 검증되고 결과가 저장되었음을 나타내는 `QueryProofVerified` 이벤트를 수신합니다 ([submit_query.ts](https://github.com/gluwa/ccnext-testnet-bridge-examples/blob/main/custom-contracts-bridging/scripts/submit_query.ts)에서 발췌):

```typescript
const queryProofVerified = PROVER_ABI.find(
  (abiElement) =>
    abiElement.type === 'event' && abiElement.name === 'QueryProofVerified'
);

const proverFilter = await ccNextPublicClient.createEventFilter({
  address: proverContractAddress as `0x${string}`,
  events: [queryProofVerified],
  fromBlock: BigInt(startBlock - BLOCK_LAG),
  toBlock: BigInt(currentBlock - BLOCK_LAG),
});

const proverLogs = await ccNextPublicClient.getFilterLogs({
  filter: proverFilter,
});

for (const log of proverLogs) {
  const decodedLog = decodeEventLog({
    abi: PROVER_ABI,
    data: log.data,
    topics: log.topics,
  });

  if (decodedLog.eventName === 'QueryProofVerified') {
    // Query proof verified, results are now available in contract storage
    console.log(`Query proving complete. QueryId: ${decodedLog.args.queryId}`);
  }
}
```

**4단계: 컨트랙트 스토리지에서 결과 읽기**

컨트랙트는 `UniversalSmartContract_Core`를 상속하고 prover 컨트랙트 스토리지에서 `ResultSegment[]`를 읽습니다 ([MintableUSCBridge.sol](https://github.com/gluwa/CCNext-smart-contracts/blob/main/contracts/MintableUSCBridge.sol)에서 발췌):

```solidity
import {ICreditcoinPublicProver} from "@gluwa/creditcoin-public-prover/contracts/sol/Prover.sol";
import {UniversalSmartContract_Core} from "./UniversalSmartContract_Core.sol";

abstract contract MintableUSCBridge is UniversalSmartContract_Core, ERC20 {
    mapping(bytes32 => bool) public processedQueries;

    function mintFromQuery(
        address proverContractAddr,  // Prover contract address required
        bytes32 queryId              // Query ID from prover contract
    ) external {
        if (processedQueries[queryId]) revert QueryAlreadyProcessed();
        processedQueries[queryId] = true;

        // Read ResultSegment[] from prover contract storage
        (bytes32 functionSig, ResultSegment[] memory eventSegments) =
            _processUSCQuery(proverContractAddr, queryId);

        // Extract data from segments (not full transaction receipts)
        if (bytes4(functionSig) != TRANSFER_EVENT_SIG) revert InvalidFunctionSignature();

        address from = address(uint160(uint256(bytes32(eventSegments[0].abiBytes))));
        address to = address(uint160(uint256(bytes32(eventSegments[1].abiBytes))));
        uint256 amount = uint256(bytes32(eventSegments[2].abiBytes));

        // Process extracted segments
        require(to == address(0), "Invalid burn address");
        _mint(from, amount);

        emit TokensMinted(address(this), from, amount, queryId);
    }
}
```

`_processUSCQuery` 함수는 prover 컨트랙트 스토리지에서 읽습니다 ([UniversalSmartContract_Core.sol](https://github.com/gluwa/CCNext-smart-contracts/blob/main/contracts/UniversalSmartContract_Core.sol)에서 발췌):

```solidity
function _processUSCQuery(
    address proverContractAddr,
    bytes32 queryId
) internal returns(bytes32 functionSignature, ResultSegment[] memory eventSegments) {
    ICreditcoinPublicProver prover = ICreditcoinPublicProver(proverContractAddr);
    QueryDetails memory queryDetails = prover.getQueryDetails(queryId);
    ResultSegment[] memory resultSegments = queryDetails.resultSegments;

    // Extract function signature and event segments from stored results
    require(resultSegments.length >= 8, "Invalid result length");
    functionSignature = resultSegments[4].abiBytes;

    uint256 resultLength = resultSegments.length;
    eventSegments = new ResultSegment[](resultLength - 5);
    for (uint256 i = 5; i < resultLength;) {
        eventSegments[i] = resultSegments[i];
        unchecked { ++i; }
    }
}
```

#### V2 접근 방식:

v2에서는 전체 트랜잭션 데이터와 함께 직접 precompile 호출로 이 프로세스가 정말 단순화되었습니다 ([SimpleMinterUSC.sol](https://github.com/gluwa/usc-testnet-bridge-examples/blob/main/contracts/sol/SimpleMinterUSC.sol)에서 발췌):

```solidity
import {EvmV1Decoder} from "./EvmV1Decoder.sol";
contract SimpleMinterUSC is ERC20 {
    INativeQueryVerifier public immutable VERIFIER =
        INativeQueryVerifier(0x0000000000000000000000000000000000000FD2);

    mapping(bytes32 => bool) public processedQueries;

    function mintFromQuery(
        uint64 chainKey,
        uint64 blockHeight,
        bytes calldata encodedTransaction,
        bytes32 merkleRoot,
        INativeQueryVerifier.MerkleProofEntry[] calldata siblings,
        bytes32 lowerEndpointDigest,
        bytes32[] calldata continuityRoots
    ) external returns (bool) {
        //Calculate transaction index from merkle proof
        uint256 transactionIndex = _calculateTransactionIndex(siblings);
        bytes32 txKey = keccak256(abi.encodePacked(chainKey, blockHeight, transactionIndex));
        require(!processedQueries[txKey], "Query already processed");

        //Build proofs and verify directly
        INativeQueryVerifier.MerkleProof memory merkleProof =
            INativeQueryVerifier.MerkleProof({root: merkleRoot, siblings: siblings});
        INativeQueryVerifier.ContinuityProof memory continuityProof =
            INativeQueryVerifier.ContinuityProof({
                lowerEndpointDigest: lowerEndpointDigest,
                roots: continuityRoots
            });

        bool verified = VERIFIER.verifyAndEmit(
            chainKey, blockHeight, encodedTransaction, merkleProof, continuityProof
        );
        require(verified, "Verification failed");

        processedQueries[txKey] = true;

        //Decode and validate receipt status (full transaction receipt available)
        EvmV1Decoder.ReceiptFields memory receipt =
            EvmV1Decoder.decodeReceiptFields(encodedTransaction);
        require(receipt.receiptStatus == 1, "Transaction did not succeed");

        //Extract Transfer events from full receipt logs
        bytes32 TRANSFER_EVENT_SIGNATURE = 0xddf252ad1be2c89b69c2b068fc378daa952ba7f163c4a11628f55a4df523b3ef;
        EvmV1Decoder.LogEntry[] memory transferLogs =
            EvmV1Decoder.getLogsByEventSignature(receipt, TRANSFER_EVENT_SIGNATURE);
        require(transferLogs.length > 0, "No transfer events found");

        //Process transfer events from full receipt
        EvmV1Decoder.CommonTxFields memory txFields =
            EvmV1Decoder.decodeCommonTxFields(encodedTransaction);
        bool found = _processTransferLogs(transferLogs, txFields.from);
        require(found, "No valid burn transfer found");

        _mint(msg.sender, MINT_AMOUNT);
        emit TokensMinted(address(this), msg.sender, MINT_AMOUNT, txKey);

        return true;
    }
}
```

## 문제 해결

### 일반적인 문제

**1. "Proof not found" / Proof API 서버에서 404**

**문제:** 증명 생성 API 서버가 트랜잭션 또는 블록을 찾을 수 없음

**해결 방법:**

* proof 서버가 실행 중인지 확인: `/api/v1/health`
* 트랜잭션 해시/인덱스가 올바른지 확인
* 소스 체인에 블록 높이가 존재하고 증명 블록이 생성되었는지 확인

**2. "Attestation not found" / "Block not ready"**

**문제:** 아직 attestor 네트워크에서 블록이 증명되지 않음

**해결 방법:**

* attestor들이 블록을 증명할 때까지 대기
* attestor 네트워크가 실행 중인지 확인하고 Creditcoin 노드에서 증명 상태 확인

**3. "Attestations missing for chain {chain_key}"**

**문제:** 지정된 `chainKey`에 대한 증명이 존재하지 않음

**해결 방법:**

* attestor 네트워크가 이 체인에 대해 실행 중이고 구성되어 있는지 확인
* 온체인에 증명이 존재하는지 확인하고 `chainKey`가 소스 체인 구성과 일치하는지 확인

**4. "Proof server down" / 연결 거부**

**문제:** 증명 생성 API 서버가 실행되지 않거나 접근할 수 없음

**해결 방법:**

* proof 서버가 실행 중인지 확인: `curl http://localhost:3100/api/v1/health`
* 서버 로그를 확인하고 문제에 대한 RPC 연결 확인

**5. "No attestation or checkpoint found after block {height}"**

**문제:** 연속성 증명에는 쿼리 블록 이후의 증명/체크포인트가 필요하지만 아직 존재하지 않음

**해결 방법:**

* 다음 증명이 생성될 때까지 대기 (증명 간격 확인, 예: 10블록마다)

**6. "Transaction hash not found"**

**문제:** Proof API가 해시로 트랜잭션을 찾을 수 없음

**해결 방법:**

* 대신 블록 높이와 트랜잭션 인덱스 사용 시도: `/api/v1/proof/{chainKey}/{height}/{txIndex}`

**7. "Query height out of range"**

**문제:** 요청된 블록 높이가 유효하지 않음

**해결 방법:**

* 높이가 하한 및 상한 증명 경계 사이에 있는지 확인 (경계에 있거나 외부에 있지 않아야 함)
* 쿼리 높이가 증명/체크포인트 블록과 일치하는지 확인 (조정이 필요할 수 있음)
* 체인에 대한 증명이 존재하고 쿼리 높이가 증명된 범위 내에 있는지 확인

**8. "트랜잭션에 대해 Proof 생성 서버가 500 오류 반환"**

문제: 증명 생성 API 서버가 증명 요청 시 HTTP 500 (내부 서버 오류) 반환

해결 방법:

* 특정 오류 세부 정보를 위해 proof 서버 로그 확인 (데이터베이스 오류, RPC 실패, Merkle 증명 생성 문제)
* 트랜잭션 블록이 증명되었는지 확인 (위의 문제 #2 참조)
* 소스 체인 RPC가 접근 가능하고 올바르게 응답하는지 확인
* proof 서버가 캐싱을 사용하는 경우 데이터베이스 연결 확인
* 트랜잭션이 소스 체인에 존재하고 완결되었는지 확인
* 증명 요청 전에 증명을 기다림 (아래 Worker Flow 참조)

**9. Worker Flow: `targetHeight` 이해하기**

문제: worker가 언제 증명을 요청해야 하고 어떤 높이를 기다려야 하는지에 대한 혼란

설명:

* `targetHeight`는 현재 소스 체인 헤드가 *아닙니다*
* `targetHeight = transactionBlockNumber + 1`
* worker는 `latestAttestation >= targetHeight`가 될 때까지 기다려야 합니다
* 이것은 증명 생성 전에 트랜잭션 블록 자체가 증명되었음을 보장합니다

이것이 중요한 이유는 트랜잭션이 블록 N에 있는 경우, 블록 N+1이 증명될 때까지 기다리면 블록 N이 증명됨을 보장하기 때문입니다. 블록이 증명되기 전에 증명을 요청하면 "Block not ready" 또는 500 오류가 발생합니다. 이것은 위의 문제 #8과 관련이 있습니다 - 조기 증명 요청이 서버 오류를 유발합니다

### 디버깅 팁

* 자세한 오류 메시지를 위해 proof 서버 로그 확인
* 모든 서비스가 실행 중인지 확인: Creditcoin 노드, attestor 네트워크, proof 서버, 데이터베이스
* proof API health 엔드포인트 사용: `/api/v1/health`
* Creditcoin RPC 또는 `Polkadot.js`를 통해 증명 상태 확인
* 더 자세한 정보를 위해 proof 서버에서 상세 로깅 활성화
